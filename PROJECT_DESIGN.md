# shiziyouxi 项目设计书

## 1. 项目概述

`shiziyouxi` 是一个面向汉字学习的小游戏集合项目。项目当前以纯浏览器版本为主线，目标是在无需安装依赖的情况下，通过轻量、有趣、可扩展的小游戏帮助用户识字、认字和熟悉汉字读音。

当前已实现的主游戏是“汉字拼图”：系统随机选择一个常用汉字，将其渲染为图片后切分成拼图块，玩家通过交换拼图块还原目标汉字。游戏完成后展示用时、步数，并调用浏览器语音能力朗读汉字。

项目同时保留 Python Flet 版本，作为早期实现、功能验证和后续桌面端维护的参考。

## 2. 项目目标

### 2.1 产品目标

- 提供一个简单、直观、适合儿童或汉字学习者使用的汉字小游戏集合。
- 通过拼图、配对、听读等互动形式增强汉字学习的趣味性。
- 优先支持浏览器直接运行，降低使用和分发成本。
- 为后续新增多个汉字小游戏预留统一入口和扩展结构。

### 2.2 技术目标

- 使用原生 HTML、CSS、JavaScript 构建浏览器版本，避免复杂构建链。
- 将共享资源、集合外壳和单个小游戏拆分清楚，方便维护。
- 每个小游戏以独立目录组织，并通过统一的 `mount` / `unmount` 生命周期接入集合外壳。
- 数据文件独立维护，浏览器版本和 Flet 版本分别使用适合自身环境的数据格式。

## 3. 当前功能范围

### 3.1 Web 游戏集合

当前 Web 版本具备以下能力：

- 打开后进入游戏选择主页。
- 主页展示可玩的小游戏卡片。
- 当前已注册“汉字拼图”游戏。
- 点击“开始游戏”后进入对应小游戏。
- 小游戏内支持返回主页。
- 公共样式、汉字数据、语音朗读能力由 `common/` 目录统一提供。

### 3.2 汉字拼图小游戏

当前汉字拼图具备以下能力：

- 随机选择目标汉字。
- 显示目标汉字和拼音。
- 支持三档难度：
  - 简单：2×2
  - 普通：3×3
  - 困难：4×4
- 使用 Canvas 渲染目标汉字。
- 按当前难度将汉字图片切分为若干拼图块。
- 随机打乱拼图块，且避免开局即完成。
- 玩家点击两个拼图块进行交换。
- 记录步数。
- 记录用时。
- 支持提示功能：临时标记当前位置正确或错误的拼图块。
- 判断是否完成拼图。
- 完成后弹窗展示：
  - 汉字
  - 拼音
  - 用时
  - 步数
- 完成后调用浏览器 Web Speech API 朗读汉字。
- 根据窗口大小动态调整拼图块大小，尽量保持画面居中且避免滚动。

### 3.3 Python Flet 备用版本

项目保留两个 Flet 实现：

- 完整版：使用 PIL 将汉字渲染为图片并切片，通过 Flet 的拖拽组件交换拼图块。
- 简化版：不做图片切片，使用彩色编号方块验证拼图交互逻辑。

Flet 版本当前不是项目主线，但可作为桌面端、跨端或后续实验功能的参考实现。

## 4. 用户与使用场景

### 4.1 目标用户

- 初学汉字的儿童。
- 正在学习中文的非母语学习者。
- 想通过小游戏熟悉常用汉字和读音的用户。
- 项目维护者或开发者，用于继续扩展更多汉字学习小游戏。

### 4.2 典型使用流程

1. 用户打开 `game.html` 或 `web/index.html`。
2. 浏览器进入游戏集合主页。
3. 用户选择“汉字拼图”。
4. 系统随机展示一个目标汉字及其拼音。
5. 用户选择或使用默认难度开始拼图。
6. 用户点击两个拼图块进行交换。
7. 用户可点击“提示”查看哪些拼图块位置正确。
8. 用户完成拼图。
9. 系统显示结果并朗读该汉字。
10. 用户选择“再来一局”或返回主页。

## 6. Web 端设计

### 6.1 页面加载设计

`web/index.html` 是当前浏览器主入口，按顺序加载：

1. 全局基础样式：`common/css/base.css`
2. 主页样式：`web/app/home.css`
3. 汉字拼图样式：`web/games/hanzi-puzzle/style.css`
4. 汉字数据：`common/js/data.js`
5. 语音能力：`common/js/speech.js`
6. 汉字拼图游戏逻辑：`web/games/hanzi-puzzle/main.js`
7. 集合外壳逻辑：`web/app/app.js`

当前项目没有使用模块打包器，因此各脚本通过全局变量和挂载到 `window` 的工厂函数进行连接。

### 6.2 集合外壳设计

集合外壳由 `web/app/app.js` 负责，核心职责包括：

- 获取页面根节点 `#app`。
- 维护当前运行中的小游戏实例 `currentGame`。
- 定义游戏注册表 `games`。
- 渲染游戏选择主页。
- 根据游戏 ID 创建并挂载对应小游戏。
- 切换游戏或返回主页时调用当前游戏的 `unmount`。

小游戏注册结构如下：

```js
{
    id: "puzzle",
    icon: "🧩",
    title: "汉字拼图",
    description: "交换拼图块，还原目标汉字。完成后会自动读出这个字。",
    create: createPuzzleGame,
}
```

### 6.3 小游戏生命周期约定

每个小游戏应提供一个工厂函数，返回统一生命周期对象：

```js
function createSomeGame({ root, onBack }) {
    function mount() {
        // 渲染界面，绑定事件，初始化游戏
    }

    function unmount() {
        // 清理计时器、事件监听、语音或临时状态
    }

    return { mount, unmount };
}
```

当前汉字拼图通过 `window.createPuzzleGame = createPuzzleGame` 暴露给集合外壳。

## 7. 汉字拼图设计

### 7.1 核心状态

汉字拼图的核心状态包括：

| 状态 | 含义 |
|---|---|
| `currentHanzi` | 当前目标汉字。 |
| `gridSize` | 当前网格大小，如 2、3、4。 |
| `tiles` | 当前拼图块排列顺序。数组中的值表示原始 tile index。 |
| `correctOrder` | 正确顺序，即 `[0, 1, 2, ...]`。 |
| `selectedTile` | 第一次点击选中的拼图块位置。 |
| `moves` | 当前步数。 |
| `startTime` | 当前局开始时间。 |
| `timerInterval` | 计时器句柄。 |
| `hanziCanvas` | 完整汉字图像 Canvas。 |
| `tileCanvases` | 切分后的拼图块 Canvas。 |
| `hintTimeout` | 提示效果自动清理计时器。 |
| `mounted` | 当前游戏是否仍处于挂载状态。 |

### 7.2 游戏初始化流程

新一局游戏流程如下：

```text
读取难度
  ↓
随机选择汉字
  ↓
显示目标汉字和拼音
  ↓
Canvas 渲染完整汉字图片
  ↓
按 gridSize 切分 Canvas
  ↓
生成正确顺序 correctOrder
  ↓
复制并打乱 tiles
  ↓
如果打乱后仍为正确顺序，则重新打乱
  ↓
重置步数、选中状态和时间
  ↓
渲染拼图网格
  ↓
启动计时器
```

### 7.3 Canvas 渲染与切片

汉字拼图不依赖图片资源，而是在浏览器中实时使用 Canvas 绘制汉字。

渲染策略：

- 创建固定大小的正方形 Canvas。
- 绘制白色背景。
- 使用蓝色粗体 CJK 字体绘制目标汉字。
- 添加轻微阴影增强视觉效果。
- 使用 `drawImage` 将完整 Canvas 按网格切片。

切片顺序为行优先：

```text
index = row * gridSize + col
```

玩家完成游戏的判定也基于该顺序。

### 7.4 拼图交互设计

当前 Web 版采用点击交换交互：

1. 第一次点击某个拼图块，将其标记为选中。
2. 第二次点击另一个拼图块。
3. 若两次点击位置不同，则交换 `tiles` 中两个位置的值。
4. 步数加一。
5. 重新渲染拼图。
6. 检查是否完成。

该交互相比拖拽更适合移动端和触屏设备，也减少了浏览器拖拽兼容性问题。

### 7.5 胜利判定

胜利条件：

```text
tiles 与 correctOrder 完全一致
```

当前实现通过 `tiles.join() === correctOrder.join()` 判断。

胜利后：

- 停止计时器。
- 计算总耗时。
- 填充结果弹窗内容。
- 显示结果弹窗。
- 调用 `speakHanzi(currentHanzi, pinyin)` 朗读汉字。

### 7.6 提示设计

提示功能用于降低挫败感，尤其适合儿童学习场景。

点击提示后：

- 遍历当前所有拼图块。
- 如果某块的原始 index 等于当前位置，则标记为正确。
- 否则标记为错误。
- 正确块显示绿色视觉效果和 `✓`。
- 错误块显示红色视觉效果和 `✗`。
- 2.5 秒后自动清除提示状态。

### 7.7 响应式设计

Web 版目标是尽量适配桌面和移动端。

主要策略：

- 页面使用 `100dvh` 适配移动浏览器动态视口。
- 主容器居中显示。
- 使用 `clamp()` 控制间距、字体和按钮大小。
- 拼图块大小根据容器宽度和视口高度计算。
- 拼图块大小限制在 44px 到 100px 之间。
- 小屏幕下游戏标题区改为单列布局。

## 8. 数据设计

### 8.1 数据来源

项目使用 `common_3500.txt` 保存常用汉字原始数据，说明中定位为《通用规范汉字表》一级字表，共 3500 字。

### 8.2 数据生成

`generate_data.py` 使用 `pypinyin` 生成拼音映射，并输出 `hanzi_data.json`。

生成流程：

```text
读取 common_3500.txt
  ↓
逐字调用 pypinyin 获取带声调拼音
  ↓
过滤无法生成拼音的字符
  ↓
生成 chars 数组和 pinyin 映射
  ↓
写入 hanzi_data.json
```

`hanzi_data.json` 结构：

```json
{
  "chars": ["一", "乙", "二"],
  "pinyin": {
    "一": "yī",
    "乙": "yǐ",
    "二": "èr"
  }
}
```

### 8.3 浏览器数据

浏览器版本使用 `common/js/data.js`，提供：

- `HANZI_LIST`
- `PINYIN_MAP`

这些变量供小游戏直接读取。

### 8.4 Flet 数据

Flet 版本直接读取项目根目录下的 `hanzi_data.json`。

## 9. 语音设计

浏览器语音能力由 `common/js/speech.js` 提供。

核心逻辑：

- 检查浏览器是否支持 `speechSynthesis`。
- 每次朗读前先取消上一段朗读。
- 创建 `SpeechSynthesisUtterance`。
- 设置语言为 `zh-CN`。
- 设置朗读速度为 `0.85`。
- 调用 `window.speechSynthesis.speak()`。

该能力当前用于汉字拼图完成后的自动朗读，后续也可用于听音辨字、汉字跟读等小游戏。

## 11. Flet 版本设计

### 11.1 完整版 Flet

`flet_app/main.py` 是完整 Flet 实现，主要特点：

- 使用 PIL 生成汉字图片。
- 使用 PIL 按网格切分图片。
- 使用 Flet `Draggable` / `DragTarget` 实现拖拽交换。
- 支持难度选择、计时、步数、提示、胜利弹窗。
- 依赖系统 CJK 字体；缺少字体时视觉效果会退化。

### 11.2 简化版 Flet

`flet_app/main_simple.py` 是简化实现，主要特点：

- 不依赖 PIL 图片生成。
- 使用彩色编号方块模拟拼图块。
- 使用点击选择再点击交换的交互。
- 适合快速验证游戏规则和基础 UI。

### 11.3 Flet 版本定位

Flet 版本当前定位为：

- 早期原型保留。
- 桌面端可能性的技术储备。
- 与 Web 实现对比的参考代码。
- 非当前主线开发入口。

## 12. 扩展设计

### 12.1 新增小游戏目录规范

后续新增小游戏建议使用以下结构：

```text
web/games/<game-name>/
  main.js
  style.css
```

例如新增“汉字配对”：

```text
web/games/hanzi-match/
  main.js
  style.css
```

### 12.2 新增小游戏接入步骤

1. 在 `web/games/<game-name>/` 创建 `main.js` 和 `style.css`。
2. 在 `main.js` 中实现 `createXxxGame({ root, onBack })`。
3. 返回 `{ mount, unmount }`。
4. 在 `web/index.html` 中引入该游戏的 CSS 和 JS。
5. 在 `web/app/app.js` 的 `games` 数组中注册游戏。
6. 如果需要共享能力，优先放入 `common/`。

## 14. 当前约束与注意事项

- Web 版本当前使用全局变量组织代码，没有模块系统。
- 浏览器 Web Speech API 的中文语音质量依赖用户系统和浏览器环境。
- Canvas 汉字渲染依赖系统字体；不同平台显示可能略有差异。
- 当前打乱算法使用随机排序，简单但存在偏差。
- 当前只记录单局步数和时间，没有持久化成绩。
- 当前没有自动化测试。
- 当前 Flet 版本与 Web 版本分开维护，功能不要求完全同步。

## 16. 维护原则

后续维护建议遵循以下原则：

- 当前主线优先维护 Web 版本。
- 新小游戏必须独立放在 `web/games/<game-name>/`。
- 共享数据和能力放入 `common/`，避免复制到每个小游戏。
- 集合外壳只负责游戏选择和切换，不写具体游戏逻辑。
- 单个小游戏应在 `unmount` 中清理自己的计时器、事件监听和临时状态。
- Flet 版本作为备用实现独立维护，不应阻塞 Web 版本迭代。
